# 荣耀手表 5 接入说明

## 目标

把当前原型接到荣耀手表 5 真机，实现手表端直接控制“红树河熊蓝牙车钥匙”。

## 当前已完成的抽象

当前仓库已经把下面三层拆开了：

1. `src/core`
   负责协议格式、CRC 校验、挑战应答、安全会话状态机。
2. `src/adapters`
   负责具体的 BLE 读写实现，当前已包含 `MockBleAdapter` 和 `HonorWatch5BleAdapter` 骨架。
3. `src/watch`
   负责手表端交互流和动作分发。

这样做的好处是，后续接荣耀手表 5 时，主要替换 `BleAdapter` 的平台实现，不需要重写协议和业务逻辑。

## 真机接入建议

### 方案 A：手表直连 BLE 车钥匙

适用于荣耀手表 5 的应用运行环境提供 BLE Central / GATT Client 能力时。

要做的事：

- 新建一个荣耀手表 5 应用壳工程
- 用荣耀手表 5 实际开放的蓝牙 API 填充 `HonorWatchBlePlatform`
- 把 `WatchKeyController` 接到手表 UI 页面
- 为危险动作增加二次确认和生物认证

推荐映射：

- `platform.scan()` -> 平台 BLE 扫描 API
- `platform.connect()` -> 平台 GATT 连接 API
- `HonorWatch5BleAdapter.write()` -> 向控制特征值写入编码帧
- `HonorWatch5BleAdapter.read()` -> 消费通知特征值里的响应帧

建议先在应用壳侧准备以下信息：

- 车钥匙 BLE 服务 UUID
- 控制写入特征 UUID
- 响应通知特征 UUID
- 扫描过滤规则，例如设备名前缀或厂商数据标识

### 方案 B：手表作为前端，手机作为 BLE 中继

如果荣耀手表 5 第三方运行环境对 BLE 权限或后台连接有限制，这个方案更稳。

职责拆分：

- 手表端只负责 UI、身份确认、指令发起
- 手机端 App 负责真实的 BLE 扫描、连接、鉴权和命令下发
- 手表与手机之间通过厂商提供的设备互联能力同步指令

这个方案在商业落地里通常更容易过审，也更利于后台保活。

## 安全建议

- 不要把最终量产密钥直接硬编码在手表端
- 优先使用一次性挑战应答，而不是明文固定口令
- 对 `unlock`、`trunk`、`panic` 增加短时有效签名和超时机制
- 给每条控制指令加入时间戳、防重放随机数和设备绑定标识
- 真机版本至少需要额外一层系统级身份校验

## 下一步最值得做的事

1. 明确“红树河熊蓝牙车钥匙”的真实 GATT 服务 UUID、特征 UUID 和认证流程
2. 确认荣耀手表 5 第三方应用环境是否开放 BLE Central 能力
3. 根据结果选 “手表直连” 或 “手表 + 手机中继” 架构
